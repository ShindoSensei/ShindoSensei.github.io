Roborent

A web app that connects residential landlords directly with prospective tenants, without the presence of agents. It primarily targets expatriates.

Live Site

 Github Link

Why

Landlord/tenants currently pay a hefty commission when engaging an agent. Yet, a lot of an agent's work, namely sourcing properties and paperwork, can be replaced by code. Roborent aims to reduce costs for both the landlord and tenant, improving the overall rental ecosystem.

How

  • Stack & Tools: Ruby on Rails, Javascript, Bootstrap, PostgreSQL, Google Maps API, Cloudinary, Twitter Typeahead Searchbar Gem
  • Architecture: Model-View-Controller (MVC) framework
  • Deployment: Heroku
  • Process: This was a 1 week group project built in collaboration with 2 of my friends Xavier & Gek Teng. We first wireframed using a web app called Moqup and also sketched out an Entity Relationship Diagram (ERD). Thereafter, we split the work according to pages and used Github as a version control to manage workflow. As team lead of this project, I was in charge of coordinating workstreams and managing the git master branch in preparation for deployment.

Challenges

This was the first collaborative project we each experienced. The main difficulty was not the individual parts but rather, how we could manage the workload as a team and minimise merge conflicts. To tackle this, for the first 1-2 days, we sat next to each other and managed each feature merge together. Over time as we grew more comfortable, we were able to merge confidently and communicate remotely. Skype's share screen feature was helpful when resolving merge conflicts too!

Moving Forward

  • Add filters to shortlists to allow member to look through properties easily
  • Add more Google Map API features - e.g. Nearby shopping malls,gyms,etc.
  • Fix searchbar styling (When typing, it currently looks like it's filled in when it's not)

CrowdSub

A web app that crowdsources subtitle editing of foreign films. It was built with Viki.com as an example by consuming its API for data.

Live Site

 Github Link

Why

One day, my girlfriend was watching a korean drama on Viki and came across an english subtitle that was a bit too literal. This is a common problem in fan-subbed foreign shows, as fans are usually not professional translators. Being the good boyfriend that I am, I delved into the contribution structure of Viki in an attempt to solve this issue.

I discovered that for each of Viki's video content, there was on average 1-2 final editors. So I thought, why not crowdsource the final editing of subtitles too? CrowdSub aims to fill this void by having an upvote system, allowing the community to float the most appropriate subtitle edits to the top and alerting final editors of the most pressing improvements required.

How

  • Stack & Tools: Node.js(Express framework), Javascript, jQuery(with AJAX), Bootstrap, MongoDB(NoSQL), Viki API with Crypto authentication
  • Architecture: Model-View-Controller (MVC) framework
  • Deployment: Heroku
  • Process: This was a 1 week solo project. I began by sketching wireframes using paper and a pencil. Thereafter, I broke down the project into 3 small parts - Understanding and consuming the Viki API, creating the front page and finishing off the CrowdWall page.

Challenges

This was the first project I did involving API consumption, Node.js & AJAX (for partial rendering). To tackle these, I designated sufficient time for each part and did some extensive research before I began coding. In addition, I focused primarily on a MVP to prevent feature creep - a dangerous thing to have considering my first foray into the aforementioned fields.

Moving Forward

  • Upgrading the site to a live version using socket.io (as opposed to AJAX).
  • Repackaging any collected CrowdSubbed comments into an API, allowing editors to easily consume this data.

Pulse

An IOT-Web app that assists military commanders in monitoring the heart rate of their men, notifying them of anyone in danger.

Live Site

 Github Link

Why

Believe it or not, Pulse was initially built to target gym instructors, enabling them to tell if, for instance, a cycling class was pushing themselves! After speaking to friends with UX experience, gym instructors and my girlfriend(an omniscient and avid gym-goer), my team figured that experienced instructors need only look at the faces of the class to tell if it was time to rev up the exercise.

We decided that instead of focusing on training intensity, why not focus on training safety? That was when we fixated on the military - since the Singapore Armed Forces(SAF) is notorious for their focus on safety.

How

  • Stack & Tools: Ruby On Rails, Arduino(IOT with cloudMQTT), React.js, Javascript, jQuery(with AJAX), Bootstrap,Font Awesome, PostgreSQL
  • Architecture: Model-View-Controller (MVC) framework
  • Deployment: Heroku
  • Process: This was a 10 day project built in collaboration with my friends Iskandar and John. We began by sketching wireframes of the website and the Entity Relationship Diagram on paper. We also mapped out the structure of the React components. The Github kanban board was helpful to spread the work around. John, the IOT maverick, focused on building up the MQTT framework whilst Iskandar and myself built the API backend to prepare it for IOT synchronisation and front end consumption. Upon completing the IOT and backend portions, we proceeded onto the front end and split the work up by pages. We built each of our own parts in git feature branches, merging onto a common development branch only after each feature was stable.

Challenges

This was the first full-scale project we did with React.js. Working with React was difficult at first as we had to switch our thinking from a traditional front-end approach to a state-based approach. In addition, Rails' synchronous nature proved difficult to integrate with MQTT IOT which initially caused the app to hang when it was called. To solve this, we ran Rails Active Jobs and sent the MQTT job to the background, allowing users to move on with other app functions.

Moving Forward

  • To implement Rails' Action Cable in place of the current AJAX interval polling framework
  • To implement React Routing such that there is a unique link for each footer button (as they are all currently at '/trainings')
  • To add React typecheck validators
  • To re-organise the code such that server side code is not all in the trainings#index controller method
  • To improve responsive styling
  • To add heart rate graphs of each trainee, providing commanders with a real-time update of their status, akin to an ECG heart monitor, using the react-chartjs package.
